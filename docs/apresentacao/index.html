CardioRemoto02
O contexto

Acompanhar quem não pode esperar a próxima consulta

O Hospital Universitário acompanha pacientes cardiovasculares por meio de agentes de saúde em campo. Cada visita coleta dados antropométricos, sinais vitais e exames — mas o trabalho acontece onde a conectividade é instável.

  • Dados precisam ser coletados mesmo sem internet
  • É preciso saber quem está em risco e quem visitar primeiro
  • Tudo deve convergir para um banco central quando houver rede

Controlado

Moderado

Sem dados

A ideia — do documento de requisitos

Um app de coleta que guia a decisão clínica

O agente de saúde faz login, vê a lista de pacientes e, para cada um, pode cadastrar, registrar visitas e acompanhar a evolução dos exames. O sistema classifica o risco e ordena por prioridade de visita automaticamente — funcionando offline e sincronizando quando possível.

CardioRemoto05
Regras de negócio — definidas no documento

Como o risco, a prioridade e os alertas são calculados

🎯 Metas

  • PA < 140/90
  • HbA1c < 7%
  • LDL < 130 mg/dL

📅 Prioridade de visita

  • Controlado: a cada 90 dias
  • Não controlado: a cada 30 dias
  • Atrasados têm prioridade

⚠️ Alertas críticos

  • PA ≥ 180/120 ou < 90/60
  • Glicemia ≥ 250 ou < 70
  • LDL ≥ 190 mg/dL

Essas regras vivem em um único módulo compartilhado — a mesma lógica roda no app e no backend.

CardioRemoto09
Sincronização — o coração do produto

Automática, invisível e sem perder dados

  • Toda alteração local entra na fila de sincronização
  • Push envia as mutações pendentes; o servidor faz merge por updatedAt (last-write-wins)
  • Pull incremental por cursor traz só o que mudou
  • Dispara sozinha: ao salvar, ao abrir o app e ao reconectar
  • Puxar para atualizar força um sync; um toast avisa o que chegou
  • Offline nunca bloqueia: os dados ficam seguros até haver rede

CardioRemoto19
Próximo passo · 1

🔐 Autenticação & usuários

Hoje

Login local contra um agente semeado no dispositivo; bloqueio por tentativas já implementado.

Para produção

  • Serviço de auth (OAuth2 / JWT)
  • Cadastro e gestão de agentes
  • Perfis e permissões (RBAC)
  • Recuperação de senha e MFA

Por quê: dados de saúde exigem identidade forte e rastreabilidade de quem coletou o quê.

CardioRemoto20
Próximo passo · 2

🗄️ Banco de dados & API

Hoje

Banco central em node:sqlite (arquivo/memória) e um servidor node:http mínimo — prova de conceito do contrato de sync.

Para produção

  • PostgreSQL gerenciado + backups/replicação
  • API versionada e autenticada
  • Validação server-side (Zod compartilhado)
  • Resolução de conflitos por campo, idempotência, paginação

Por quê: durabilidade, concorrência e escala com muitos agentes e dispositivos.

CardioRemoto21
Próximo passo · 3

🛡️ Segurança & LGPD

Hoje

Rede local permitida (ATS), dados em texto no dispositivo. Suficiente para protótipo, insuficiente para dados reais.

Para produção

  • Criptografia do banco local (SQLCipher)
  • TLS + gestão de segredos
  • Consentimento e trilha de auditoria
  • Anonimização e política de retenção

Por quê: são dados sensíveis de saúde — a LGPD exige proteção em repouso e em trânsito.
